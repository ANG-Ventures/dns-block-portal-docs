TL;DR

When AdGuard Home or NextDNS blocks a domain, the user currently sees NextDNS's static "blocked" page (or a generic ERR_CONNECTION_RESET, depending on the layer). There is no way to self-service unblock without SSHing into the gateway or asking Forge to run a command.

This project replaces that experience with a first-party block page hosted on the LAN that offers three actions per block:

  1. Unblock on this device — 5min / 1hr / permanent. AGH per-client rule.
  2. Unblock everywhere — 5min / 1hr / permanent. AGH + NextDNS allowlist.
  3. Explain — show which blocklist / CNAME chain caught it.

All unblocks are logged to JSONL + announced to Discord #alerts in real time, with a daily digest to #adguard-setup and a rate-limit-tripwire alert for suspicious activity.

Goals

  • Primary: "I clicked a link, got blocked, and within 10 seconds I'm on the real page" — no SSH, no agent intervention, works from Ace's phone in the kitchen.
  • Secondary: Audit trail — every unblock is captured with who / what / when / where / why-blocked, so we can spot patterns (e.g., a single device whitelisting 20 trackers in a day → malware suspect).
  • Tertiary: Pave the way for proper internal TLS across the homelab via step-ca (HA UI, Frigate UI, this portal, etc. all served with trusted certs).

Non-goals

  • Not a NextDNS replacement. NextDNS remains the blocklist source of truth.
  • Not a public-internet service. Portal binds to LAN only.
  • Not a generic admin panel. No "edit blocklists," no "see all queries." Those live in AGH / NextDNS native UIs.
  • Not for Kitty's devices in Phase 1 (her devices won't have the root CA installed initially; she'll see cert warnings but still hit the page).

Why use a DNS proxy and front-door

NextDNS does not let us customize its block page beyond toggling it on/off. AGH can rewrite domains to a portal, but doing that directly would require moving blocklist authority into AGH, which contradicts the current architecture.

The proxy and front-door keep the responsibilities clean:

  • AGH remains the LAN entrypoint and client-identity layer.
  • NextDNS remains the blocklist and security authority.
  • The selected blockpage proxy only changes responses that NextDNS has already classified as blocked.
  • The HA N100 front-door chooses proxy vs direct NextDNS and fails open when the proxy is unhealthy.
  • If the custom portal path fails, DNS resolution survives and users fall back to stock NextDNS behavior.
  • Routine project/doc/build updates do not post to #announcements; DNS operational transitions still post to #alerts, and script errors still post to #logs.
TLS requirement

For HTTPS sites, the browser believes it is connecting to the original blocked hostname, e.g. store.yeelight.com, even though DNS points that hostname to the portal. A self-signed certificate for blocked.local.ace does not produce a clean UX; it produces a scary warning for the original domain.

Good UX requires an internal CA trusted by the devices that should see the portal. The portal then serves dynamic certificates whose SAN matches the blocked hostname.

  • Build phase: code can run with a self-signed cert for local smoke tests.
  • Real MVP: internal CA + trusted device enrollment for Ace's primary devices.
  • Fallback: if cert work is incomplete, blocked sites still fall back to NextDNS's standard block page or a browser warning depending on route.

Browser extensions (Option C in the design discussion) are explicitly out-of-scope because they don't work on iOS Safari, iOS Chrome, or iOS Brave (all forced to WebKit and disallowed from extending it). The portal must work on mobile, so it has to be a real page.

Per-device unblock semantics

NextDNS allowlist is global — there is no NextDNS-native way to allow a domain for one device. Per-device unblock is therefore an AdGuard Home concern:

  1. Portal receives "unblock this device" click.
  2. Resolves client source IP → AGH persistent-client name (e.g. 192.168.1.45 → aces-iphone). Persistent clients are configured in the AGH UI; this list must be kept current.
  3. Adds rule: @@||store.yeelight.com^$client=aces-iphone to AGH custom rules via the local-dns skill.
  4. NextDNS continues to block the domain for every other device — AGH answers first for the iPhone only.
  5. If duration ≠ permanent, schedule auto-removal at T+duration.

If the client IP doesn't resolve to a persistent client, the portal blocks device-scoped self-service unblock and asks for the device to be added as an AGH persistent client first. Explicit IP/CIDR rules remain possible as an operator-only emergency path, but the portal should not silently rely on volatile DHCP IPs.

"Unblock everywhere"

Calls local-dns allow <domain> --scope both:

  • Adds @@||<domain>^ to AGH custom rules.
  • Posts to NextDNS allowlist API (POST /profiles/13b946/allowlist).
  • AGH cache flush.

Wildcards are rejected by NextDNS; portal pre-validates and shows an inline error if the user tries to allow *.tracker.com.

Resilience Plan

Incident that triggered this section

2026-05-15: HA N100 was powered off. AGH at 192.168.1.208 disappeared from the LAN. UniFi DHCP pointed clients at .208 first, and Tailscale DNS pointed tailnet clients only at .208. MBP and ACE-PC lost DNS until Tailscale and DHCP were temporarily moved to direct NextDNS/public fallback. After HA N100 powered back on, .208 answered again and normal DNS was restored.

Design rule

DNS can lose filtering, but it must not lose resolution. In a failure, clients should fall back to NextDNS's standard block page or unfiltered emergency DNS, not total outage.

Required steady-state
  • UniFi DHCP DNS: 192.168.1.208,45.90.28.215.
  • Tailscale DNS: 192.168.1.208,45.90.28.215.
  • AGH upstream after promotion: HA N100 DNS front-door 192.168.1.208:5354.
  • Front-door degraded resolver: direct NextDNS DoH profile 13b946.
  • No single powered-off homelab box can be the only DNS resolver for a critical client path.
  • Portal/proxy services are never enabled automatically from this repo; live routing is a separate manual gate.
Preflight before any live DNS change
  1. ping 192.168.1.208 succeeds.
  2. dig @192.168.1.208 google.com A succeeds.
  3. dig @192.168.1.208 doubleclick.net A returns blockpage.nextdns.io.
  4. dig @45.90.28.215 google.com A succeeds.

  • Frigate/proxy down: AGH should use direct NextDNS fallback upstream.
  • NextDNS down: only then consider temporary public emergency fallback (1.1.1.1) and announce that filtering is degraded.
  • Portal down: DNS should still resolve; blocked domains should fall back to NextDNS blockpage behavior, not outage.

Resilience system options

Alerts are useful, but insufficient: if DNS is broken, Ace may not be able to reach Discord to see the alert. The system must take corrective action or provide a resolver path that does not depend on AGH being alive.

R0 — Current baseline

Status: implemented.

  • UniFi DHCP advertises 192.168.1.208,45.90.28.215.
  • Tailscale DNS advertises 192.168.1.208,45.90.28.215.
  • Direct NextDNS fallback exists if clients honor the second resolver.

Risk: many clients do not fail over cleanly or quickly from resolver #1 to resolver #2. This reduces blast radius but does not guarantee usability.

R1 — Automatic control-plane failover

Status: deferred.

A small watchdog runs somewhere that does not depend on AGH DNS for outbound access. If AGH fails two checks in a row:

  1. Set Tailscale DNS to 45.90.28.215,1.1.1.1.
  2. Set UniFi DHCP DNS to 45.90.28.215,1.1.1.1.
  3. Announce degraded mode.
  4. Keep checking AGH.
  5. After AGH passes N consecutive checks, restore 192.168.1.208,45.90.28.215.

Pros: fast for Tailscale clients, low implementation complexity, easy rollback.

Cons: DHCP clients may not update until renewal or network reconnect. This is failover automation, not a perfect HA resolver.

Current decision: do not build this yet. The implemented path is fallback plus transition alerting. Mutating UniFi DHCP and Tailscale DNS automatically is a stronger intervention than we need until we see repeated failures that client fallback does not absorb.

R2 — Secondary LAN resolver

Status: likely worth doing.

Run a minimal resolver/proxy on Frigate N100 as a second local DNS endpoint. DHCP/Tailscale can advertise:

  • Primary: AGH 192.168.1.208
  • Secondary: Frigate resolver 192.168.1.176
  • Tertiary or direct fallback: NextDNS 45.90.28.215

The Frigate resolver should forward to NextDNS directly and should not depend on AGH.

Pros: keeps LAN DNS local even when HA N100 is down.

Cons: client resolver failover is still inconsistent; also requires keeping Frigate powered.

R3 — Router/front-door resolver

Status: possible but higher risk.

Make UDM-SE (192.168.1.1) the only DNS server advertised to clients, and configure its dnsmasq to forward in strict order:

  1. AGH / portal path
  2. Frigate resolver
  3. NextDNS direct

Pros: clients only know one stable resolver, the gateway. Failover happens server-side instead of relying on client resolver behavior.

2026-05-16 Future Goal — Dual Blockpage Hosts

This section is superseded by the 2026-05-17 HA100-primary specification pivot below. Dual-host mode remains useful only if Frigate stays configured as a fallback/reference after HA100-primary is proven.

The idea: keep Frigate N100 as the practical first blockpage host, but leave a clear path for HA N100 to become a second compatible blockpage host later. Once both hosts can run the same portal/proxy contract, we can choose which one is active.

North star
AGH on HA N100
  -> DNS front-door on HA N100
      selected blockpage host:
        option A: Frigate N100 192.168.1.176
        option B: HA N100 192.168.1.208
      resolver fail-open:
        direct NextDNS DoH profile 13b946

The important part is that direct NextDNS remains the safety valve. Switching between Frigate and HA N100 changes where the custom blockpage lives. It does not replace resolver-level fail-open.

Why this is appealing

Frigate is the right MVP host because it is ordinary Ubuntu, has systemd, has Python, and has a clean path to binding :443. It is the fastest way to ship the product.

HA N100 is attractive later because AGH already lives there. If the DNS control plane and a backup blockpage host can both live there cleanly, the system gets a nicer operational story:

  • Frigate can go down without immediately removing every custom blockpage option.
  • We can intentionally move the active blockpage host for maintenance.
  • The front-door can grow into a real policy router instead of a one-off proxy selector.
  • A future dashboard could show: active host, standby host, health state, last switch, and rollback button.
Operating modes
Mode 1: Frigate primary

This was the pre-pivot ship path. It remains staged as fallback/reference.

AGH -> HA front-door -> Frigate proxy/portal -> NextDNS

If Frigate is healthy, blocked domains resolve to the Frigate portal IP. If Frigate is unhealthy, the front-door fails open to direct NextDNS. Users lose the custom blockpage temporarily, but DNS survives.

Mode 2: HA N100 primary

Future compact mode.

AGH -> HA front-door -> HA-hosted proxy/portal -> NextDNS

This reduces cross-host dependency but couples the custom blockpage to HA N100. If HA N100 dies, clients still need DHCP/Tailscale fallback to direct NextDNS.

Mode 3: Selectable active host

Future resilient mode.

AGH -> HA front-door -> selected blockpage host
                     -> Frigate or HA N100

This should start as manual selection, not automatic failover. A person chooses frigate or ha_n100, the front-door applies that setting, and it posts a state-change alert. Automatic host failover can come later only after we prove DNS caching, TLS trust, audit continuity, and unblock consistency.

    portal_ip: 192.168.1.208

The active host has to be visible and reversible. Silent switching is dangerous because clients cache DNS answers, certificates are tied to host behavior, and unblocks/audits must land in the expected place.

Both hosts need to behave like the same product:

  • same /health shape
  • same /tls-allow behavior

  • same explain behavior

  • same alert destination semantics
  • same persistent-client naming
  • same NextDNS block detection
  • same dry-run/live safety gates
  • same root CA/certificate trust story, or an explicitly documented per-host CA story

The cleanest future version uses one internal CA and identical portal behavior on both hosts. The safer intermediate version treats HA N100 hosting as a lab mode until cert handling and shared state are fully tested.

Security model

Trusted vs untrusted sources

HAProxy uses trusted_portal_sources = [192.168.1.0/24] with allow_broad_trusted_sources: true. This trusts every LAN device for portal access, covering both:

  • Direct-LAN access (e.g. https://blocked.local.ace/ from your MacBook on .88): arrives at HAProxy with the device's real IP.
  • Hairpin access (blocked-domain sinkhole → WAN IP → UDM hairpin): arrives with src=192.168.1.1 (UDM hairpin SNAT collapses identity).

Both paths land in the /24, so every LAN client sees the portal. IoT devices don't browse blocked content; granular per-device trust would just add ops burden (deprecated M4 follow-up).

A WAN-vantage client (cellular, third-party probe) arrives with its real public IP preserved (UDM does NOT SNAT inbound DNAT) — that IP is not in 192.168.1.0/24 → HAProxy reject_unknown_sni. Verified 2026-05-27 with check-host.net 6-node probe across IL/NL/RS/FR/US/BG/KZ/CA/CN/ID.

Portal mutation auth

/unblock and /reblock require Authorization: Bearer <DNS_ADMIN_SHARED_SECRET> (stored in 1Password as `DNS Portal Admin Auth

  • Shared Secret`). Pre-Wave-3 the deployed container predated this hardening; rebuilt + redeployed 2026-05-27.

Browser-session admin path also exists via /admin/login form + CSRF token, for interactive use.

Cert chain

Ace Local Root CA (Phase 5C custody) → Ace Local Intermediate 2026 → per-SNI leaf minted by Caddy on demand. Trusted devices that installed the root see green padlock on portal.

Custody (L1, shipped 2026-05-27): the canonical PKI lives in 1Password (DNS Portal Local PKI (root+int), Engineering vault). Live disk copy at /share/dns-portal/caddy/pki/{ace-root.crt, ace-int.crt, ace-int.key} is monitored every 5min by multivantage-probe check E (SHA256 fingerprint vs pinned value). Drift triggers a Discord alert with the one-line recovery command:

bash ~/Projects/dns-block-portal/deploy/inject-pki.sh --apply

Wipe-and-restore is non-disruptive: trusted devices keep their green padlock because the root cert SHA256 is unchanged. Rotation playbook at ~/Projects/dns-block-portal/docs/longterm/L1-rotation.md (intermediate rotation: ~15min, no downtime; root rotation: ~1wk with overlap window).

L1 — Portal Caddy cert custody

Status: ✅ SHIPPED 2026-05-27 (Wave-3, same session as everything else). Why: trusted devices visiting https://<blocked-domain>/ get a cert error unless their OS trusts the Caddy-minted leaf. Phase 5C already shipped the root-CA custody pattern for the legacy dns-blockpage add-on — L1 makes the same chain available to dns-portal AND survivable across volume rebuilds. Rotation playbook: docs/longterm/L1-rotation.md.

Today's behavior

dns-portal Caddy is configured to mint per-SNI leaves on-demand from a local PKI. The Caddyfile fragment:

pki {
  ca ace_local {
    name "Ace Local CA"
    root_cn "Ace Local Root CA"
    intermediate_cn "Ace Local Intermediate 2026"
    root {
      format pem_file
      cert /data/caddy/pki/ace-root.crt
    }
    intermediate {
      format pem_file
      cert /data/caddy/pki/ace-int.crt
      key /data/caddy/pki/ace-int.key
    }
  }
}
on_demand_tls {
  permission http http://127.0.0.1:18080/internal/is-blocked
}

When Caddy boots without ace-root.crt / ace-int.crt, it generates its own. If we wipe the data volume or rebuild the container without preserving /data/caddy, a fresh self-signed root is born and trusted devices that had the old root installed start showing cert errors.

Goal

Trusted devices that have Ace Local Root CA installed (Phase 5C custody) should see a green padlock on the portal page — no warning, no "Continue anyway" click. Untrusted devices get the warning page (existing UX, unchanged).

Approach

  1. Single canonical PKI. Generate exactly one root + intermediate, sign them as Ace Local Root CA / Ace Local Intermediate 2026 (already matches Phase 5C). Store both inside 1Password and on /share/dns-portal/caddy/pki/ (the container's bind-mounted /data/caddy).
  2. Idempotent injection. A pre-start hook (deploy/inject-pki.sh) that:
    • Reads the four files (ace-root.crt, ace-root.key, ace-int.crt, ace-int.key) from 1Password.
    • Writes them only if missing or fingerprint-mismatched.
    • chmod 600.
  3. Rebuild safety. Container image must NOT bake the PKI. Rebuilds stay stateless; the PKI lives in the volume.
  4. Verify chain. tests/phase5e/test_pki_continuity.py would:
    • Connect to https://blocked.local.ace:9443/healthz with the root in the trust store.
    • Confirm cert validates without --insecure.
    • Confirm root fingerprint matches the 1Password reference.

Files to touch

  • deploy/inject-pki.sh (new) — runs at container ENTRYPOINT or compose pre-start.
  • deploy/Dockerfile — invoke inject-pki at entrypoint.
  • 1Password Engineering vault — new item DNS Portal Local PKI (root+int) with the four PEMs.
  • tests/phase5e/test_pki_continuity.py (new).

Rotation

Root: 10-year validity. Rotate via manual procedure documented in docs/longterm/L1-rotation.md (TBD): generate new root, distribute to trusted devices, swap intermediate.

Intermediate: 1-year validity. Rotate automatically via a launchd job. Caddy keeps the old leaf certs valid until expiry, so a brief skew during rotation is invisible to users.

L1 PKI rotation playbook

When and how to rotate the DNS Portal PKI. Companion to: docs/longterm/L1-portal-cert-custody.md (the design).

When to rotate

Intermediate (1-year validity)
  • Routine: every 9-10 months (before expiry).
  • Triggered by: cert-healthcheck alert "cert_expiring_soon" (30-day warning).
  • Trusted-device impact: none — leaf certs already minted from the old intermediate stay valid until their own expiry; Caddy mints new leaves from the new intermediate after rotation.
Root (10-year validity)
  • Routine: every 8-9 years (before expiry).
  • Triggered by: cert-healthcheck alert OR catastrophic key exposure.
  • Trusted-device impact: high — each trusted device must install the new root before the old one expires, OR they all see cert errors simultaneously. Plan a rollout window.

Rotation procedure (intermediate only)

# 1. Generate new intermediate signed by the existing root.
#    (Run on a trusted workstation with the root key — i.e. NOT HA100.)
cd ~/Projects/dns-block-portal
bash scripts/phase5c/rotate-intermediate.sh    # TODO: ship this script
# Writes new ace-int.crt + ace-int.key locally.

# 2. Verify the new intermediate validates against the existing root.
openssl verify -CAfile /tmp/ace-root.crt /tmp/ace-int-new.crt

# 3. Update 1Password.
op item edit l5x6g5d6ehtujm5t2jn4e2ljaq \
  "intermediate_crt[text]=$(cat /tmp/ace-int-new.crt)" \
  "intermediate_key[password]=$(cat /tmp/ace-int-new.key)" \
  "intermediate_fingerprint_sha256[text]=$(openssl x509 -in /tmp/ace-int-new.crt -noout -fingerprint -sha256 | cut -d= -f2)"

# 4. Update the pinned fingerprint in code.
#    - scripts/multivantage-probe.py    PINNED_ROOT_FP (unchanged for int rotation)
#    - tests/phase5e/test_pki_continuity.py    PINNED_INT_FP (update)

# 5. Force-push to HA100 (overwrites old intermediate).
bash deploy/inject-pki.sh --force --apply

# 6. Verify portal serves leaves signed by the new intermediate.
echo | openssl s_client -connect 192.168.1.5:9443 -servername blocked.local.ace 2>/dev/null \
  | openssl x509 -noout -issuer
# Expected: issuer=CN=Ace Local Intermediate <new year>

# 7. Run e2e tests.
PHASE5E_E2E_LIVE=1 python3 -m pytest tests/phase5e/test_pki_continuity.py -v

Total time: ~15 min. No downtime.

Rotation procedure (root)

This is invasive. Plan ≥1 week of overlap.

# 2. Generate new intermediate signed by the new root.
openssl x509 -req -days 365 -in /tmp/ace-int-new.csr \
    -CA /tmp/ace-root-new.crt -CAkey /path/to/new-root.key \
    -out /tmp/ace-int-new.crt

# 3. Distribute new root cert to all trusted devices.
#    Use Phase 5C custody pattern. References:
#    - skills/local-dns/references/ca-trust-macos.md
#    - skills/local-dns/references/ca-trust-windows.md
#    - skills/local-dns/references/ca-trust-ios.md
#    Wait until ALL trusted devices have the new root installed AND a

#    Update tests/phase5e/test_pki_continuity.py PINNED_ROOT_FP +
#    PINNED_INT_FP. Update scripts/multivantage-probe.py PINNED_ROOT_FP.

# 5. Force-push to HA100.

# 6. e2e + manual smoke test from each trusted device.

# The spurious self-signed root and any leaf certs Caddy signed under it

L3 — HAProxy → nftables migration

Status: SPEC (2026-05-27). Speculative — HAProxy works, this is a "could we make it simpler" exercise. Why now: HAProxy is overkill for L4 SNI routing. It carries ~100MB RSS, runs uvicorn-style worker threads, and uses a 1000-line config that mostly does TCP-inspect + per-ACL backend pick. nftables can do the same in a kernel chain with zero userspace component.

Today's HAProxy role

From ha-addon/dns-sni-frontdoor/haproxy.cfg.template:

frontend phase5d_tls_frontdoor
  mode tcp
  bind {{bind_host}}:{{bind_port}}
  tcp-request inspect-delay {{inspect_delay}}
  tcp-request content accept if { req_ssl_hello_type 1 }
  acl is_mdca req_ssl_sni -i {{mdca_hostname}}
  acl is_trusted src {{trusted_portal_sources}}
  use_backend npm_mdca       if is_mdca
  use_backend portal_caddy   if is_trusted
  default_backend reject_unknown_sni

backend npm_mdca
  mode tcp
  server npm {{npm_backend}}

backend portal_caddy
  mode tcp
  server portal {{portal_backend}}

backend reject_unknown_sni
  mode tcp
  tcp-request content reject

Four ACL decisions, two real backends. nftables can do this directly via a custom kernel module... actually no, nftables can't read TLS SNI from a TCP payload without a userspace agent (it's not a netfilter primitive). So we'd need either:

  • (A) nftables + a tiny eBPF program that parses ClientHello SNI into a meta value, then nftables maps SNI → DNAT target.
  • (B) nginx stream with SNI routing — much closer in spirit to HAProxy but already runs on Linux and is smaller.
  • (C) Caddy L4 module (caddyserver/caddy-l4) which is what we evaluated and rejected at Phase 5D Gate 0.

Decision tree

Option Footprint Operational risk Effort
A — nftables + eBPF <5MB HIGH — eBPF debugging is painful, kernel version dependencies 1-2 weeks
B — nginx stream ~30MB LOW 1 day
C — caddy-l4 ~50MB MEDIUM — Phase 5D Gate 0 rejected it for fail-closed on unknown SNI 1 day (re-evaluate)

Recommendation

Don't migrate. HAProxy dns-sni-frontdoor is 100MB RSS, runs in an HA add-on container, has clean logs, fails closed correctly, and we have e2e tests covering its behavior. There's no real pain. The win from L3 is "nicer architecture diagram," not "less work" or "fewer outages."

If we ever pursue this, recommendation order: B → C → A. nginx-stream is the lowest-risk swap if we care.

Failure modes to preserve

Whatever replacement we'd pick must preserve the Phase 5D security model:

  • mdca.duckdns.org → NPM passthrough, always.
  • Trusted-source non-mdca → portal Caddy.
  • Untrusted-source non-mdca → REJECT (TCP reset, not 200).
  • WAN clients must arrive with their real public IP (no source-IP collapse) so trusted_portal_sources ACL works.

tests/phase5e/test_wave2_e2e.py::TestPortalFlow exercises all three. A migration would need those tests to pass against the new dataplane.

Done definition (if pursued)

Effort

If we pick B (nginx-stream): ~1 day including the e2e test sweep.

Dependencies

None. Independent migration.

Status: SPEC (2026-05-27). Speculative — only needed if Ace stands up a second public hostname behind the same WAN :443. Single ACL, single backend.

Multiple public hostnames behind the same WAN :443, each routed to its own backend, while still preserving the Phase 5D security model (trusted-source-only portal, untrusted reject).

Approach

Option A — list ACL (smallest change)

acl is_paperless req_ssl_sni -i paperless.angventures.io

use_backend npm_wiki       if is_wiki

acl is_public req_ssl_sni -i -f /etc/haproxy/public-snis.txt

use_backend npm_mdca      if { req_ssl_sni -i mdca.duckdns.org }
use_backend npm_wiki      if is_wiki

acl is_public req_ssl_sni -i -f /etc/haproxy/public-snis.txt

use_backend portal_caddy     if is_trusted
default_backend reject_unknown_sni

  mode tcp

Add a hostname by adding it to public-snis.txt + creating the NPM proxy host. HAProxy stays a dumb pass-through; NPM does the host-based routing. Cleanest.

  • Extra hop on every visit (30-100ms).

R1 — HMAC-signed ?domain= to close open-redirect

Without this, an attacker could craft https://blocked.local.ace/?domain=your-trusted-bank.com to social-engineer the user into allowlisting an unintended domain. Today's spec just trusts the query param.

Implementation (Option A from review — simpler):

  1. Each blocked-SNI portal Caddy response includes a redirect with both exp and t as URL params, where t is the HMAC of the full domain + exp payload (so neither can be tampered independently): 
    exp   = now_unix() + 300                     # 5-min validity
payload = f"domain:{domain}:{exp}"           # the HMAC input
t     = base64url(hmac_sha256(
          secret=DNS_ADMIN_SESSION_SECRET,
          message=payload,
        ))
Location: https://blocked.local.ace/?domain={domain}&exp={exp}&t={t}
  2. blocked.local.ace handler validates by re-deriving the HMAC from URL params, not by trusting them:
    • Extract domain, exp, t from the URL.
    • Re-compute expected_t = base64url(hmac_sha256(SECRET, f"domain:{domain}:{exp}")).
    • Constant-time compare t vs expected_t. If mismatch → 400 Bad Request: invalid redirect token.
    • Reject if int(exp) < now_unix() → 400 Bad Request: expired redirect token.
    • The exp is INSIDE the HMAC input — attacker cannot replay an old token with a future exp because the HMAC would mismatch.
  3. Direct navigation to /?domain=X without t+exp → ignore domain, render the generic admin form (no domain prefilled).

This means an attacker can NOT craft a working ?domain= URL because they don't have DNS_ADMIN_SESSION_SECRET. They cannot extend expiry on a captured token because exp is HMAC-protected. The token is single-purpose (only valid for the original SNI's redirect chain) and expires fast.

  1. Client (Brave on .88) requests https://doubleclick.net/.
  2. Brave asks OS resolver → AGH at .208:53.
  3. AGH says doubleclick.net IN A 68.185.70.45 with TTL=10s (blocked_response_ttl). Brave caches this.
  4. Brave hits .45:443 SNI=doubleclick.net → portal Caddy → block page.
  5. User clicks "Unblock everywhere, 5 min."
  6. Portal writes @@||doubleclick.net^ to AGH user_rules + calls /control/filtering/refresh + /control/cache_clear. AGH now forwards the query upstream → returns real Google IP.
  7. Portal redirects the browser to https://doubleclick.net/.
  8. Browser re-resolves doubleclick.net from its OWN cache (which still has A=68.185.70.45). Cache TTL hasn't expired yet.
  9. Browser hits .45:443 again → portal Caddy → block page reappears.

The DNS query never reaches AGH in step 8, so the @@-rule doesn't help. The browser uses its cached answer for the TTL window.

Cache layers between browser and AGH

Layer Default TTL behavior How to invalidate
Browser DNS cache Chrome respects TTL but min ~60s; Firefox 60s/120s; Brave inherits Chromium chrome://net-internals/#dns Clear host cache; or close tab + wait
OS resolver (macOS mDNSResponder) Honors TTL but has its own 0-30s minimum sudo dscacheutil -flushcache; sudo killall -HUP mDNSResponder
OS resolver (Windows) DNS Client Service caches; min ~30s ipconfig /flushdns
OS resolver (iOS/Android) Opaque; close+reopen browser is the only user-side fix Restart browser
AGH (we control) blocked_response_ttl: 10 (already low) /control/cache_clear (we already do this)
HSTS (separate issue!) If portal Caddy ever sends Strict-Transport-Security for the spoofed domain, browser locks to portal even after DNS fixes Don't send HSTS for spoofed SNIs — already correct in Caddyfile

We don't control browser/OS caches. AGH side is already optimized.

Solutions (ordered by effort + impact)

L8.1 — Lower blocked_response_ttl to 1 second (or 0)

Tells well-behaved clients to never cache the block response. Doesn't help with browser minimums but reduces the window for browsers that honor TTL.

Effort: 1-line AGH config change. Impact: small but free. Risk: more DNS query load on AGH. Negligible — AGH handles thousands of QPS without breaking a sweat.

L8.2 — Portal redirects via HTTP Cache-Control: no-store + ?_=<nonce> query string

Already setting no-store on portal responses. The browser cache for the URL is bypassed, but the DNS cache is independent of the URL cache. So this doesn't help with the symptom.

Impact: zero for this problem.

L8.3 — Portal redirects to a different URL (the real destination via a hop)

Instead of Location: https://doubleclick.net/, redirect to https://blocked.local.ace/redirect-after-unblock?target=https%3A%2F%2Fdoubleclick.net%2F&unblocked_ts=<ts>. The interstitial page sleeps 12s + auto-refreshes the target via JS location.href = .... By the time the JS fires, browser DNS cache for the target has likely expired.

Effort: 1 hour. Impact: medium. Works for most domains but feels slow (12s delay).

L8.4 — Portal POSTs a "purge browser DNS" beacon

Open hidden image to a known-non-existent subdomain of the target (?cachebust=randomstr.doubleclick.net). The browser would issue a fresh DNS query for the subdomain. Doesn't help — DNS cache is keyed per-hostname, not per-subdomain.

Better: serve the portal block page with an <iframe sandbox="..."> or a <link rel="dns-prefetch" href="//doubleclick.net"> AFTER the unblock POST returns. The dns-prefetch hint might trigger Chrome to re-resolve. Unreliable across browsers.

Effort: small. Impact: flaky. Not recommended.

L8.5 — Portal returns a 503 + Retry-After: 12 instead of 200, then JS does the redirect

If portal Caddy returns 503 Service Unavailable + Retry-After: 12 on the post-unblock visit (we can detect it via a session marker or URL param), the browser interprets this as "service not available." Some browsers will retry after the delay, others won't. Not reliable.

L8.6 — RECOMMENDED: User instruction + browser-side DNS bypass

Add to the unblock success page (currently shows "✅ unblock done"):

Your browser cached the old DNS answer. To see the real site immediately:

  • Chrome/Brave: Open chrome://net-internals/#dns → Clear host cache, then refresh.
  • Firefox: open Settings → Privacy → Cookies and Site Data → Clear Data → DNS cache, then refresh.
  • OR: wait 60 seconds, then refresh.

Auto-detect browser, show the right snippet.

Also useful: redirect target uses ?_dns_bust=<random> so the cached URL response (separate from DNS cache) is busted. This catches the case where it's an HTTP cache hit (rare since we send no-store but defense in depth).

Effort: 1 hour. Impact: high — user knows exactly what to do. UX-wise less magical than L8.3 but actually works deterministically.

L8.7 — Use DNS-over-HTTPS from the browser to AGH (skip browser DNS cache)

Configure Chrome's "Use secure DNS" to point at AGH (which doesn't support DoH natively — would need a proxy in front). Browser cache becomes much shorter when using DoH because Chrome uses a separate, smaller cache for DoH.

Effort: medium (need AGH+DoH front-end like dnscrypt-proxy). Impact: medium. Other side effects (DoH cert pinning, fallback).

L8.8 — Pre-emptive: when portal shows the block page, set a hidden cookie that says "user is about to unblock; expect short TTL." Combined with L8.3's interstitial, this makes the wait UX clear.

Bundle with L8.6.

Recommendation (revised 2026-05-27 after Ace pushback)

Drop L8.1. Lowering AGH TTL from 10s → 1s only helps OS-resolver caches that honor low TTLs (some macOS configs), saves 0-9s, doesn't move the needle on browser caches that already ignore TTLs < 60s. Not worth the config drift risk.

New issues

RR1 (minor, clarification only): The HMAC token spec says:

t = base64url(hmac_sha256(secret=DNS_ADMIN_SESSION_SECRET, message="domain:doubleclick.net:<expiry_unix>"))

…and that exp is "encoded in the token via second HMAC field or appended &exp=<unix> validated by the same HMAC." This is ambiguous about whether exp lives inside the HMAC-protected payload or is just appended to the URL. If exp is in the URL but not in the HMAC input, the attacker can replay an old expired token with a future exp.

Clarify: the HMAC input MUST be the full string "domain:<sni>:<exp>" AND the server MUST re-derive expected HMAC from domain AND exp in the URL. So exp is "in the URL AND in the HMAC input" — both. The v2 spec is probably meaning this but the prose is ambiguous. Fix by restating:

The server-side handler MUST extract domain and exp from the URL, recompute expected_t = hmac_sha256(SECRET, f"domain:{domain}:{exp}"), and constant-time compare with the URL's t. If either domain or exp is tampered, the HMAC will mismatch and the request is rejected with 400.

Recommendation: add this clarification before merging L6.1. ~5 min edit.
